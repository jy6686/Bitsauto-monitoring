The fix is live and confirmed working in production. Here's a clean summary:

What was changed
Root cause: The old recording (/var/spool/asterisk/monitor/1781008054.183826.wav) was started by Sippy/Asterisk when the channel was created — before the call was answered. So it always captured the ringback/ringing phase. When governance cut the call after the cap, it replayed that file, and clients heard the ringtone.

Three targeted changes:

1. New startMixMonitor() in ami-governance.ts
Sends Action: MixMonitor to Asterisk with the b option (record only while bridged). This is called at bridge time — meaning the recording starts exactly when the call is answered, and never earlier.

2. Governance-specific recording file in the bridge handler
Changed from re-using Sippy's pre-answer file to creating a new governance file:

Before: /var/spool/asterisk/monitor/1781008054.183826.wav (Sippy's, includes ringback)
After: /var/spool/asterisk/monitor/gov_1781008054.183826.wav (governance-started, post-answer only)
3. Silence fallback for sub-3s calls in cutVendorLeg
If a call is cut within 3 seconds of being answered, the WAV file would be near-empty. Rather than play a click or fragment, it now plays Asterisk's built-in silence/10 (10 seconds of silence) followed by a clean hangup.

Confirmed in live logs within minutes of deployment:

[ami-governance] StartMixMonitor → channel=PJSIP/sippy-endpoint-000166cf
[call-governance] #14473 MixMonitor started (post-answer)
...
[ami-governance] CutAndPlayback → file=/var/spool/asterisk/monitor/gov_1781008054.183857

Clients will now hear clean conversation audio (or silence if the call was near-instant) instead of ringback tones.
