If a test fails, the default assumption is: the Send Rate workflow above the line is wrong — not the push engine below the line. The engine is not to be modified, refactored, or "cleaned up" during certification under any circumstances.

Tariff ID | Prefix | Current Rate | Current Effective Date

Every test then follows this pattern:

Read Baseline → Apply Test Change → Verify → Restore → Verify Restore

This guarantees zero permanent impact on production pricing, regardless of what fails.

→ Validation error returned
→ NO Sippy write occurs

T7 (Prefix Collision) — second priority:

#328 - Rate Job Audit Fields + Sheet Generated Step
What & Why
The rate notification job record currently collapses "generate the Excel sheet" and "send the email" into a single emailSent flag. These are two distinct events — you can generate successfully and then fail SMTP, and without a separate flag there is no audit evidence of what was actually prepared. Additionally, six months after a send there is no proof of exactly which template version was used or what was in the attachment. Add three new audit fields and split the email pipeline into two trackable steps.

Done looks like
Sending a notification shows two separate step indicators in the job detail: Sheet Generated and Email Sent — each can succeed or fail independently
A job record stores the template version at time of send (so changes to the template after sending don't alter historical records)
A job record stores a SHA-256 hash of the Excel attachment that was generated — operators can re-derive and verify the exact file that was emailed
The Job Detail drawer in the Notifications tab shows these additional fields in the step grid and in the metadata footer
Legacy step order is preserved: Email Sent → SBC Mapping Available → SBC Updated → Updated Tariff → Violated Rules → Approval Required → Successful; Sheet Generated appears before Email Sent
Out of scope
Re-generating or re-downloading the attachment from the job record
Template versioning system (template_version is the job-time snapshot of notificationType + destinationCount, not a full version table)
Email delivery receipts from the recipient's mail server
Steps
DB + schema — Add sheet_generated boolean, template_version varchar, and generated_attachment_hash varchar to rate_notification_jobs via direct SQL and update shared/schema.ts
Backend: split generate from send — In the send action, generate the Excel buffer first and mark sheetGenerated = true, then attempt SMTP. Compute SHA-256 of the buffer and store as generatedAttachmentHash. Capture templateVersion as "${notificationType}:v${destinationCount}" at job-creation time
Frontend: step grid — Add Sheet Generated step to JobDetailDrawer between the job metadata and Email Sent; update RnJob interface with the three new fields
Frontend: metadata footer — Show template version and attachment hash (truncated to first 12 chars + "…") in the job detail footer row

Point 1 — DB Proof (Prefix fix)

The reviewer is correct to push back. Explaining 0233 as "test data" without a query is not acceptable for a production system. The verification query they suggest is exactly right:

Point 2 — "Both" segment is wrong

The reviewer is right. "Both" is architecturally harmful. The legacy model is explicit:

A product is either Retail or Wholesale — never both. If you need both, you create two distinct products.

"Both" makes every downstream system ambiguous:

Which rate card applies?
Which clients see which notification?
Which side of the P&L does it belong to?
The fix: remove "Both" from the dropdown entirely. Only Retail and Wholesale. If a product genuinely serves both markets, it becomes two products — Mega - Retail and Mega - Wholesale — and that distinction is what drives the client wizard, governance, and reporting correctly.

Point 3 — Notification Type missing source_module

The reviewer is right but this is correctly a backlog item, not a blocker. The two operational actions:

Send Rate Notification — rate sheet dispatched to client
Rate Editor — rate applied directly in Sippy from the UI
are meaningfully different to an operations team reviewing Push History. Mixing them looks like noise without context. source_module is a small addition but has real audit value. It should be tracked from day one so history doesn't lose this context retroactively.

Point 4 — Destination should store ID + Name + dialPrefix

This is the most technically important point. Storing only destination_name violates immutability:

Names change over time ("Pakistan Ufone Mobile" → "Pakistan PTCL Mobile")
Without a frozen ID reference, the history record loses its anchor
The correct design is to store all three at push time:

destination_id — FK to global_destinations (trace to catalog entry)
destination_name — frozen as it was at the moment of push
dial_prefix — already stored, good
The destination_name being stored is still valuable — it's the historical truth of what you called it at the time. But the ID gives you the ability to trace it back to the catalog if needed. This is a small schema addition but architecturally sound.

Point 5 — Product Registry needs inheritance

This is the most important strategic point and the reviewer identified it exactly right. Without inheritance, the current segment field is just a label with no enforcement — you can create Mega with segment Wholesale and there's no Mega - Retail anywhere unless someone creates it manually.

The correct model:

Create Product: "Mega" (code: MG, trunkPrefix: 5)
         ↓ auto-creates
Mega - Retail  (code: MG-R)
Mega - Wholesale  (code: MG-W)
         ↓ immediately flows to
Client Wizard · Product Rates · Rate Manager · Notifications · Governance · Reporting

This requires:

parent_product_id FK on product_registry
Auto-variant creation logic on product insert
All downstream queries to operate at the variant level
This is its own sprint, not an addendum to the current one. But the reviewer is right that without it, every new product will reopen the same development cycle manually. The segment field as currently implemented is a temporary label — the inheritance model is the permanent fix.
