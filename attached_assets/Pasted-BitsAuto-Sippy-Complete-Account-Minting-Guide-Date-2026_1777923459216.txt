1. ARCHITECTURAL HIERARCHY — HOW SIPPY WORKS {#1-architectural-hierarchy}
In the Sippy environment, an Account cannot exist without a Customer anchor. The hierarchy is strictly enforced:

Sippy Instance (System Administrator)
│
├── Tariff (i_tariff)                    ← Rate/pricing definition
│     Defines: per-minute rates, billing intervals, connection fees
│     Required: BEFORE creating a Customer (customer inherits tariff)
│
├── Service Plan (i_billing_plan)        ← Commercial billing rules
│     Defines: subscription fees, included minutes, overage rates
│     Required: WHEN creating an Account
│
├── Routing Group (i_routing_group)      ← Traffic routing policy
│     Defines: which vendors handle calls, route selection logic
│     Required: WHEN creating an Account
│
├── Customer (i_customer)                ← The "DNA Source" / Parent Anchor
│   │  Has: name, web_login, api_password, i_tariff, i_time_zone
│   │  Has: credit_limit, balance, max_sessions, contact info
│   │  Requires: i_tariff (links pricing), web_password, name
│   │
│   ├── Account (i_account)              ← SIP endpoint / carrier line
│   │     Has: username, authname, voip_password, web_password
│   │     Has: i_billing_plan, i_routing_group, all 54 fields
│   │     Inherits: i_customer (parent anchor)
│   │
│   ├── Account
│   └── Account
│
├── Customer (i_customer=29 "RTST")      ← Our active parent
│   ├── Account 87 "Hayder"
│   ├── Account 91 "Faisal-test"
│   ├── Account 133 (our golden test v3.08)
│   └── Account 134 (our golden test v3.08b)
│
└── Customer (i_customer=1 "ssp-root")   ← System root (our API auth identity)

Key Relationships
Customer → Account: One-to-many. Every Account MUST specify i_customer.
Customer → Tariff: Each Customer requires an i_tariff at creation time. The tariff defines the rate table the customer's accounts inherit.
Account → Service Plan: Each Account requires an i_billing_plan (service plan). This is DIFFERENT from the customer's tariff.
Account → Routing Group: Each Account requires an i_routing_group.
The "Toxic Zone" — Commission Agent IDs 2–28
Per Gemini's architectural analysis: Customer IDs 2–28 were created during initial system setup and have corrupted Commission Agent associations. When creating new customers, the Commission Agent field should be set to "None" to bypass this toxic zone entirely. Our RTST customer (i_customer=29) is safely beyond this range.

Creation Order (Dependency Chain)
1. Tariff      → createTariff (or use existing)
2. Customer    → createCustomer (requires i_tariff)
3. Account     → createAccount (requires i_customer, i_billing_plan, i_routing_group)

You cannot skip steps. An Account cannot be created without a Customer. A Customer cannot be created without a Tariff.

CRITICAL NOTE: Customer Creation is Missing from BitsAuto
BitsAuto currently has NO UI to create a new Sippy Customer from scratch.

The system only imports/mirrors customers that already exist in Sippy. If a new customer (like a new wholesale partner) needs to be onboarded:

Someone must manually create them in the Sippy web panel, OR
We need to build a Customer Creation page in BitsAuto (backend routes already exist — see §7)
This is the single biggest gap in the system.

3. PHASE 2: AUTHENTICATION & SCOPING {#3-phase-2-authentication}
Two Authentication Levels
Level	User	Password	Scope	Used For
System Root	SIPPY_API_USERNAME	SIPPY_API_PASSWORD	Full system — can create accounts under ANY customer	BitsAuto's current approach
Customer-Level	Customer's web_login (e.g. RTST)	Customer's API password	Scoped to that customer ONLY — i_customer auto-injected	IchiBaan's approach
BitsAuto's Current Auth: System Root
We authenticate as ssp-root (i_customer=1) using environment secrets
This gives us full access to create accounts under any customer
We must explicitly specify i_customer in every createAccount call
This is the correct approach for a platform managing multiple customers
Alternative: Customer-Scoped Auth (IchiBaan Method)
Authenticate as the customer (e.g. RTST with password !chibaan1)
The system auto-scopes all requests to i_customer=29
Simpler but limits operations to one customer at a time
Requires "Allow API Calls" checkbox to be enabled on the customer
Authentication Method
Protocol: HTTP Digest Authentication (RFC 2617)
Transport: XML-RPC over HTTPS
Both levels use the same Sippy API URL (SIPPY_API_URL env secret)
4. PHASE 3: THE GOLDEN PAYLOAD — createAccount (54 FIELDS) {#4-phase-3-golden-payload}
This is the exact, verified payload that successfully created accounts on our Sippy instance. Every field listed here is REQUIRED — removing any single field results in a 400 "Parameter X is required" error.

THE i_time_zone BUG — Root Cause of All 501 Failures
Fact	Detail
Bug:	i_time_zone=1 causes a 501 "Fatal error" when ALL address fields are present
Scope:	Instance-specific — IchiBaan uses tz=1 on THEIR instance without issues
Root cause:	Server-side Python crash in our Sippy instance
Fix:	Use i_time_zone=370 (maps to US/Central / Europe/Andorra)
Evidence:	7 tests with tz=1 failed; 2 tests with tz=370 succeeded (see §12)

BOOLEAN / INTEGER FIELD RULE
Sippy expects <int>0</int> or <int>1</int> for all boolean-like fields. Do NOT send JavaScript true/false or XML <boolean>. The serializer must convert all booleans to integers before building the XML payload.

11. KNOWN TRAPS & LANDMINES {#11-known-traps}
Trap 1: i_time_zone = 1 Crashes Our Instance
What: Sending i_time_zone=1 with all address fields present causes a 501 Fatal Error
Why: Server-side Python crash in our specific Sippy instance
Fix: Always use i_time_zone=370 (US/Central)
Note: IchiBaan's instance handles tz=1 fine — this is instance-specific
Trap 2: Boolean Fields Must Be Integers
What: Sippy rejects <boolean>true</boolean> and JavaScript true/false
Fix: Always serialize booleans as <int>0</int> or <int>1</int>
Where: The serializer (sippy-xmlrpc-serializer.ts) handles this, but the carrier-sync.service.ts builds some boolean fields before the sweep runs
Trap 3: Commission Agent Toxic Zone (IDs 2–28)
What: Customer IDs 2–28 have corrupted Commission Agent associations
Fix: When creating customers, always set Commission Agent = "None"
Note: Our RTST (i_customer=29) is safely past this zone
Trap 4: i_tariff vs i_billing_plan Confusion
What: createCustomer requires i_tariff (rate plan). createAccount requires i_billing_plan (service plan). These are DIFFERENT fields pointing to different entities.
Fix: Don't confuse them. A tariff must exist before a customer. A service plan is assigned per-account.
Trap 5: listServicePlans Does Not Exist
What: There is no bulk-list method for service plans on our instance
Fix: Must probe individual IDs with getServicePlanInfo(i_billing_plan=N) or maintain a static list
Current known plans: 1 (sippy_support_bft), 3 (Test-1-SP), 7 (Testing-ServicePlan-30Apr)
Trap 6: Carrier Sync Stores But Doesn't Pass Contact Fields
What: The CreateCarrierPage form collects email, phone, address etc. and stores them in the DB, but carrier-sync.service.ts NEVER includes them in the Sippy payload
Fix: Add all contact/address fields to the Sippy payload builder
Trap 7: FROZEN Files Require Unfreeze Before Editing
What: carrier-sync.service.ts (SHA d73b67107ccf) and sippy-xmlrpc-serializer.ts (SHA d7421dae9b0a) are frozen
Fix: Must explicitly unfreeze before making any changes to these files
