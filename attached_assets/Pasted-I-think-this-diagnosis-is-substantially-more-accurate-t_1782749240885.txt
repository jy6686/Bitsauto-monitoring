Revised Root Cause Analysis

The platform actually has three independent destination catalogues that have become mixed together.

Catalogue	Purpose	Current Status	Should be Source Of
global_destinations	Canonical commercial catalogue	✅ Already standardized	Everything commercial
destination_product_rates	Product-specific rates	✅ Mostly clean, but duplicated because of duplicate destination IDs	Pricing
Sippy Destination Sets / destination_sets_cache	Technical routing catalogue	⚠ Legacy Sippy names (PAK Mobile MOBLIN, TELNOR, etc.)	Routing only

That separation explains why your SQL shows clean names while the UI still shows "PAK Mobile MOBLIN".

The UI isn't reading from the wrong table accidentally—it is using the wrong catalogue for the wrong purpose.

Overall assessment

I think this changes the roadmap in an important way:

Deduplicate the canonical destination catalogue first (highest priority). This eliminates ambiguous destination identity and prevents inconsistent foreign-key relationships.
Separate commercial and routing catalogues by sourcing all commercial UI components (including Rate Manager pickers) from global_destinations, while keeping Sippy naming only as technical aliases.
Introduce alias mapping and normalized-prefix uniqueness so future vendor imports and Sippy synchronizations cannot recreate the duplicate problem.
Build the commercial governance workflow (commercial_status, approvals, review queue) on top of the cleaned catalogue.

This shifts the project from what initially looked like a massive renaming exercise into a focused master data management (MDM) initiative. The canonical naming standard is already in place; the remaining work is to establish a single authoritative destination identity, preserve technical aliases for interoperability, and ensure every commercial, routing, and analytics component references that canonical record consistently.
