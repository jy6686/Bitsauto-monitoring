2. SIPPY HIERARCHY: CUSTOMER → ACCOUNT {#2-sippy-hierarchy}
Sippy Instance
├── Customer (i_customer)          ← "RTST" = i_customer=29
│   ├── Account (i_account)        ← SIP endpoint / carrier
│   │   ├── Auth Rules
│   │   └── CDR records
│   ├── Account
│   └── Account
├── Customer
│   └── ...
├── Service Plans (i_billing_plan) ← Tariff / rate plan
├── Routing Groups (i_routing_group)
└── Tariffs (i_tariff)             ← Required for customer creation

Key relationship: Every Account belongs to exactly one Customer. When creating accounts, you must specify i_customer to place the account under the correct parent.

BitsAuto's current approach:

Carrier workflow (/carriers/create): Creates a BitsAuto carrier → syncs to Sippy as an Account (under i_customer=1 root). Does NOT create a Sippy Customer.
Phase 3 workflow (/security/customer-accounts/.../create): Creates accounts under an existing mirrored customer. Requires the customer to already exist in BitsAuto DB with a sippy_customer_id.
What's missing: There is NO UI page to create a new Sippy Customer from scratch. Customers currently only exist through the Sippy import/mirror process. If a user needs to create a brand-new customer in Sippy (not import an existing one), there is no path.

3. CREATEACCOUNT — GOLDEN PAYLOAD (54 FIELDS) {#3-createaccount-golden-payload}
This is the exact payload that successfully created i_account=134 on our Sippy instance. Every field listed here is REQUIRED by our instance — removing any single one results in a 400 "Parameter X is required" error.

CRITICAL BUG DISCOVERED: i_time_zone
i_time_zone=1 causes a 501 "Fatal error" when ALL address fields are present.

This is a server-side bug specific to OUR Sippy instance
IchiBaan team uses i_time_zone=1 successfully on THEIR instance
The fix: use i_time_zone=370 (US/Central)
The existing account on our instance also uses i_time_zone=370

BOOLEAN FIELD RULE
Sippy expects integer 0 or 1 for all boolean-like fields. Do NOT send JS true/false or XML <boolean>. Always use <int>0</int> or <int>1</int>.

4. CREATECUSTOMER — PROBED REQUIREMENTS {#4-createcustomer-requirements}
The createCustomer method exists on our Sippy instance. Iterative probing revealed these required fields (in order):

Order	Field	Type	Notes
1	i_tariff	int	Tariff ID — required FIRST. Must be a valid tariff on the instance.
2	name	string	Customer name
3	web_password	string	Web portal password
?	...	...	Probing stopped here to avoid accidentally creating a customer
Important: createCustomer requires i_tariff (NOT i_billing_plan). This is a different field than what createAccount uses. A tariff must be created first (via createTariff — which our carrier-sync already does).

BitsAuto's current customer-push code (sippy-customer-write.adapter.ts) already handles:

Duplicate checking via listCustomers
Building customer payload
Push to Sippy via createCustomer
Read-back confirmation
But there is NO frontend page to trigger customer creation. The customer routes exist (POST /api/v1/customers → creates locally, POST /api/v1/customers/:id/push-to-sippy → pushes to Sippy), but no UI page walks through the full customer creation + Sippy sync flow.
