What the Legacy Screenshot Actually Shows
This is the Rate Notification Template list from the live legacy system at 23.106.59.17:8081. Key observations:

1. One client, multiple products — simultaneously

A single client (Bwt, Ebexlimited, Voortex-Ai) has templates for multiple products at once. This is not a 1:1 client→product relationship. It's M:N.

2. The legacy product taxonomy is richer than the current one

Legacy products include:
  Standard, Premium
  First Class - Wholesale
  Business Class - Wholesale
  Special Bravo - Retail
  Special Charlie - Wholesale
  Special Charlie - Retail
  Special Delta - Retail
  Economy Class - Retail
  A2Z, India Special 2/3, Idea Special

The current platform has 4 products (FC, BC, SB, SC). The legacy has 18+, with Retail/Wholesale variants of the same class. This is a much more granular taxonomy.

3. "Rate Notification Template" ≠ "Send Rate"

These are two different things:

Rate Notification Template: a record saying "client X is on product Y, use this template when notifying them"
Send Rate: the actual act of pushing a rate to Sippy tariff
The legacy system separates them. The current platform conflates them.

What This Means for the Proposed Workflow
The document proposes:

Client Wizard → Select Product(s) → Create "Pending Initial Rate Setup" Job → KAM Completes → Approval → Notification → Activate

This is correct and matches the legacy model. But it's important to note what it adds vs. what exists:

Component	Exists in current platform?
Client Wizard	✅ Exists
Product selection in wizard	✅ Exists (single product)
Multi-product selection in wizard	❌ Not yet
"Initial Client Rate Jobs" queue	❌ New queue needed
KAM rate assignment step	❌ Not built
Manager approval step	❌ Exists generically, not for rates
Rate Notification Template auto-creation	❌ Not built
Activation lifecycle (Pending → Live)	❌ Not built
This is a significant scope expansion beyond what we were certifying.

The Tension With "Certify First"
The proposed workflow is the right long-term design. But it introduces a question for the Send Rate certification:

Current Send Rate pushes rates to accounts that are already provisioned and already have tariffs. It doesn't care about the commercial lifecycle. It's an operations tool for updating existing rates.

The new workflow is about initial rate activation — a new client going live for the first time. Different use case.

These are two separate problems:

Problem A (existing accounts, rate updates):
  Send Rate → Push updated rate to 50 FC clients
  → Needs: Product Membership + bulk push
Problem B (new client onboarding, initial rates):
  Client Wizard → Initial Rate Job → KAM → Approval → Activate
  → Needs: The entire workflow described in the document

The Send Rate certification (T4A → T8 → T1 → T5) only covers Problem A. The document's proposed workflow covers Problem B. Both are needed — but they're separate builds.

My Recommendation (Discussion Only)
Don't merge them before certification.

Certification (now):
  Prove Problem A works end-to-end
  T4A → T6 → T7 → T8 → T1 → T2 → T3 → T5
After certification, two parallel workstreams:
  Stream 1: Product Membership Management (governance tool for existing accounts)
  Stream 2: Initial Client Rate Jobs (new client onboarding workflow)
These share:
  - The product taxonomy
  - The "client → product" relationship
  - The Sippy push engine
But they serve different moments in the client lifecycle.

One Critical Question for the Discussion
The legacy system has Retail vs Wholesale variants of the same product class (e.g., "Special Charlie - Wholesale" vs "Special Charlie - Retail"). The current platform doesn't have this distinction.

Before building either workstream, this needs a decision:

Does the new platform adopt Retail/Wholesale variants?
Or does it use a simpler model (FC, BC, SB, SC without retail/wholesale split)?
That product taxonomy decision affects the Client Wizard, the Send Rate product picker, and the Rate Notification Template structure. It's a foundational question worth answering before building either stream.

I think this is where the architecture should go, and it actually solves the Service Plan problem we've been discussing.
