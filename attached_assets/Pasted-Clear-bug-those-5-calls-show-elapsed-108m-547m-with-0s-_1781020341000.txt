Here's what was wrong and what was fixed:

Root cause — missed bridge events on reconnect

The governance engine only starts a cut-timer when Asterisk fires a BridgeEnter AMI event. Those events fire exactly once, when a call is first bridged. If the server restarts or AMI drops and reconnects, those events never replay — so any call already in progress at that moment gets no timer. Those 5 calls (108m–547m elapsed, all showing "0s remaining") were bridged before the last reconnect and were simply never given a countdown.

What was changed in ami-governance.ts:

Added a raw frame listener system so response events from AMI actions can be collected
Emit 'connected' event when login succeeds — triggers reconciliation
Added fetchActiveBridges() — sends CoreShowChannels AMI action and collects each bridged channel's name, duration, and bridge peer, resolves in ≤5s
What was changed in routes-call-governance.ts:

scheduleGovernedCallCut() now guards against double-scheduling (if a timer is already active, skip silently)
New reconcileActiveCalls() function: fetches all active Asterisk bridges → groups by bridge ID → matches each pair against enabled rules → for any match with no active timer, creates/reuses a DB record and schedules a cut at max(5s, capSec - elapsedSec) — so a call already 547 minutes past its cap gets cut in 5 seconds
Wired to amiGovernance.on('connected', ...) — runs 2s after every AMI login
60-second watchdog — catches any calls that slip through between reconnects
