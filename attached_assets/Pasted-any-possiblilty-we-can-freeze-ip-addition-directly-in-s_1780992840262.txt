any possiblilty we can freeze ip addition directly in sippy?

Yes. There are several ways to effectively freeze direct IP additions in Sippy, ranging from simple operational controls to hard technical enforcement.

Option 1 — Remove Sippy Permissions (Fastest)

Create Sippy roles where nobody except 1–2 super admins can access:

Customers
  → Authentication Rules
  → Add Authentication Rule
  → Delete Authentication Rule

Everyone else uses BitsAuto.

Pros
Simple
No coding
Immediate
Cons
Super admin can still bypass
Option 2 — Sippy Drift Protection (Recommended)

Treat BitsAuto as the master.

Every 5 minutes:

BitsAuto Auth Rules
       ↓
Compare
       ↓
Sippy Auth Rules

If someone adds:

IP: 1.2.3.4

directly in Sippy:

Not Found In BitsAuto
       ↓
Unauthorized Change
       ↓
Alert

Options:

Alert only
Auto-remove
Require approval

This is how enterprise configuration management systems work.

Most secure.

      ↓

Option 4 — Hide Authentication Pages in Sippy

If your team rarely needs Sippy auth pages:

Disable or block access to:

account_auth_rules.php

and related auth pages.

Users can:

View CDRs
View Routes
View Reports

but cannot reach:

Authentication

This requires Sippy UI customization and may need re-applying after upgrades.

Option 5 — Firewall/XML-RPC Enforcement (Best Long-Term)

Only BitsAuto is allowed to make authentication changes.

BitsAuto
    ↓ XML-RPC
Sippy

Human users:

Read Only

Authentication APIs are only callable from:

BitsAuto Server IP

This is the cleanest architecture.
