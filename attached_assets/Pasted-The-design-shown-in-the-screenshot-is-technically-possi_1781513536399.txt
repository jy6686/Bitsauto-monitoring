What the Diagram Is Showing

Current Mode (what you use today):

Caller ↔ Asterisk ↔ Sippy ↔ Vendor
          RTP passes through your servers

Benefits:

Call Governance can monitor calls.
Call recording works.
AMI bridge events work reliably.
RTP statistics are visible.
Vendor leg can be cut and replay/recording actions can occur.
Billing reconciliation is easier.

Target Mode (Direct RTP / Media Bypass):

Caller → Asterisk/Sippy (SIP signaling only) → Vendor

Caller ↔ Vendor
      RTP direct

Benefits:

Lower bandwidth usage.
Lower RTP load on Asterisk/Sippy.
Reduced server resources.
Slightly lower latency.

Before Implementing

I would first verify:

Does the terminating vendor support symmetric/direct RTP?
Is NAT involved?
Do you still need call recordings for compliance or dispute resolution?

If the answer to #3 is yes, keep RTP through Asterisk/Sippy for governance traffic. If the answer is no and you only need cut + silence, then Direct RTP for governance routes is a valid option.

Proposed Design

Normal Traffic

Continue using the existing flow:
Caller → Asterisk → Sippy → Vendor
RTP remains proxied through the platform
No changes to current routing, billing, recordings, or analytics.

Call Governance Bypass Traffic

Use a dedicated Governance Prefix (separate from normal traffic).
When a call matches this Governance Prefix:
Route through a dedicated Asterisk context.
Enable Direct RTP / Media Bypass.
SIP signaling remains under Asterisk/Sippy control.
RTP flows directly between Caller and Vendor.
New Governance Rule Type
cap_and_bypass

Behavior:

Call matches Governance Prefix.
Asterisk routes call through the Governance Bypass context.
RTP is established directly between Caller and Vendor.
AMI timer starts normally.
When cap_sec is reached:
Vendor leg is disconnected.
Caller remains connected.
Silence is played (or dead air is maintained).
Call Governance logs the action.
No recording is required for this rule type.
