Sippy billing increment — no API support built:

Sippy Push — What Must Change
Today a push sends:

{ prefix: "9233", rate: 0.0250 }

After destination enrichment, a push should send:

{
  prefix: "9233",
  rate: 0.0250,
  init_time: 60,       // billingIncrementInitial in seconds
  inc_time: 1,         // billingIncrementFollowing in seconds
  min_time: 60         // minimumDuration in seconds
}

Sippy's tariff rate entry XML-RPC supports init_time, inc_time, min_time. These are currently unused by the platform. The push engine (pushRateToSippy, setSippyRateEntry) needs to carry these fields through — they come from the destination catalog, not from user input.

Increment Change Workflow (future-proof)
The reviewer's scenario:

Today:    Pakistan UFONE Mobile → 9233 → 60/1 → effective 2025-01-01
Regulator changes:
Next month: Pakistan UFONE Mobile → 9233 → 30/6 → effective 2026-08-01

The billingIncrementStartDate field handles this at the destination level. The platform needs to evaluate which increment is active based on today's date:

SELECT billingIncrementInitial, billingIncrementFollowing
FROM global_destinations
WHERE id = ?
AND (billingIncrementStartDate IS NULL OR billingIncrementStartDate <= NOW())
ORDER BY billingIncrementStartDate DESC
LIMIT 1

For now: a single billingIncrementStartDate on the destination row is sufficient (no history table needed). When a regulator change comes, the admin updates the record with the new increment and its effective date. The platform picks it up automatically from that date forward. A history table (destination_increment_history) can be added later without breaking anything.
