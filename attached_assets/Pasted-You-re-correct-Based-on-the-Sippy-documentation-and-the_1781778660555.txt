You're correct. Based on the Sippy documentation and the XLSX export/import workflow you've shown, Sippy's native workflow is Excel/XLSX upload, not direct CSV upload by users.

The important distinction is:

What Sippy Users See

They export/import:

internal-ptcl_Rates.xlsx

with columns:

Action
Id
Prefix
Country
Interval 1
Interval N
Price 1
Price N
Forbidden

Expiration Date

and upload the Excel file back through the Sippy UI.

What BitsAuto Is Doing Internally

BitsAuto is currently generating a CSV payload because:

Sippy's upload backend ultimately processes tabular data.
The upload-token / upload endpoint often accepts CSV-formatted content.
It's easier programmatically to generate one row than build an XLSX workbook.

But from a telecom operations perspective, your point is valid:

If Sippy's proven working path is XLSX, then BitsAuto should eventually generate the exact same XLSX structure that Sippy exports.
