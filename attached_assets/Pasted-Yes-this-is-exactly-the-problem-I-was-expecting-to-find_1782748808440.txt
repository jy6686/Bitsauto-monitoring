Before we touch Vendor Rate Manager

I would freeze the commercial destination catalogue and perform a full governance audit.

The audit should identify:

Country abbreviations (PAK vs Pakistan)
Duplicate commercial names
Duplicate prefixes
Legacy operator names (Mobilink, Warid, etc.)
Misspellings
Truncated names
Case inconsistencies
Multiple names pointing to the same destination
Prefixes assigned to multiple destinations

Once the First Class catalogue is approved and locked, then:

Vendor imports compare against approved destinations only.
Client rate sheets always use the canonical names.
LCR uses the same destination IDs.
Margin analysis uses the same destination IDs.
Sippy publishing uses the same destination IDs.

That gives BitsAuto a single commercial source of truth instead of every module maintaining its own interpretation of destination names.

I think this is the right foundation. Building Vendor Rate Manager before cleaning the destination catalogue would force every downstream feature to compensate for inconsistent master data, whereas governing the catalogue first gives every subsequent module a clean, stable reference.
