During our earlier discussions, we agreed on implementing several security controls within the sippy, but it appears that this initiative was paused and has not been revisited since.

The primary objective of these controls was to prevent unauthorized or manual changes within sippy and ensure that all provisioning activities are performed exclusively through the BitsAuto platform.

The proposed security controls included:

•⁠  ⁠IP Addition Freeze
•⁠  ⁠Authentication Freeze
•⁠  ⁠Manual Rate Change Freeze
•⁠  ⁠Configuration Change Monitoring

The expected behavior was:

1.⁠ ⁠If any IP is added manually in sippy, the system should automatically detect and remove it within a predefined interval (e.g., 15–60 seconds), unless it was provisioned through the BitsAuto platform.
2.⁠ ⁠If any authentication is created, modified, or removed manually in sippy, the system should automatically revert the change and restore the approved configuration.
3.⁠ ⁠If any tariff or rate changes are made directly in sippy, those changes should be automatically rolled back. Only rates pushed through the BitsAuto platform should remain active.
4.⁠ ⁠Any unauthorized modification should generate immediate alerts through:
    * Email Notifications
    * WhatsApp Notifications
    * Audit Logs

The purpose of these controls was to:

•⁠  ⁠Eliminate manual configuration drift.
•⁠  ⁠Ensure sippy remains synchronized with BitsAuto.
•⁠  ⁠Maintain a single source of truth for provisioning and pricing.
•⁠  ⁠Prevent accidental or unauthorized changes by operational staff.
•⁠  ⁠Improve security, compliance, and change tracking.

We need to review why this project was put on hold, what technical or operational concerns led to the decision, and whether there are any blockers preventing implementation.

Please provide feedback on the original reasons for pausing this initiative and whether we should proceed with implementation as part of the platform security roadmap.

This initiative was not paused because the idea was wrong. It was paused because, during the Sippy integration discovery phase, we found several architectural and operational risks that needed to be solved first.

Original Objective

The objective was correct:

BitsAuto = System of Record (Source of Truth)

Sippy = Execution Engine

No human should make operational changes directly in Sippy.

All provisioning should originate from BitsAuto.

That principle is still valid and aligns with enterprise telecom OSS/BSS design.

After reviewing the Sippy documentation structure and the API capabilities, I believe the original decision should be revisited.

The key finding is that Sippy exposes management APIs for most of the objects we wanted to govern:

Authentication Rules Management (addAuthRule, updateAuthRule, etc.)
Environment and IP management APIs (list/create/update/delete environment-related configuration)
Tariff and Tariff Rates Management APIs
Account, Customer, Routing and other provisioning APIs

This changes the assessment.

The original design principle was:

Sippy = Execution Layer
