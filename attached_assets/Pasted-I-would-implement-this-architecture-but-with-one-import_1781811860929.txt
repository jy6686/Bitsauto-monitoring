I would implement this architecture, but with one important condition:

The new approach is much better than rebuilding XLSX from XML-RPC

Your manual testing already proved:

Download Tariff from Sippy
↓
Modify rate
↓
Upload same file
↓
Sippy updates correctly

While the reconstruction approach keeps hitting issues:

XML-RPC
↓
Missing Country
Missing hidden fields
Missing Sippy-specific metadata
↓
Generate new XLSX
↓
Sippy accepts upload but ignores update

So:

Download Original XLSX
↓
Patch Existing Row(s)
↓
Re-upload Same XLSX

is the correct long-term design.
