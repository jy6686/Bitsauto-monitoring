I agree with most of this analysis. Based on what you've found, I'd adjust the implementation roadmap in a few important ways.

The biggest finding

The assumption that global_destinations required a massive naming cleanup was incorrect.

Your SQL shows the canonical catalogue is already standardized:

global_destinations ≈ 151k rows
Almost every row already follows the commercial naming convention:

Pakistan Fixed
etc.
Only a handful of ISO-style names exist.

That means the destination catalogue is not the source of the naming inconsistency.

Revised priority

Based on your findings, I would prioritize the work as follows:

Deduplicate global_destinations and consolidate all foreign-key references (highest priority because it affects data integrity).
Normalize all import pipelines so every incoming prefix resolves to a single canonical destination_id.
Update the Rate Manager and related UIs to display names from global_destinations instead of Sippy-derived labels.
Implement the commercial approval workflow using the existing commercial_status field.
Enforce uniqueness and naming validation (normalized prefixes, alias registry, validation rules) to prevent future duplicates.

This sequence addresses the underlying data model first. Once there is exactly one canonical destination_id per real-world destination, the UI, rate management, reporting, routing, and governance layers all become deterministic and much easier to maintain.
