Mixing them into one screen will become unmanageable at scale. Rate Manager stays as-is. Send Rate becomes a separate workflow.

3. Per-Destination Effective Date in the Queue
Each queue row should store its own effectiveDate/effectiveTime, not a shared global field. Telecom teams routinely stagger:

Pakistan Mobile  →  0.0265  →  19-Jun 13:30 GMT
UAE Mobile       →  0.0310  →  20-Jun 00:00 GMT
Saudi Landline   →  0.0220  →  22-Jun 08:00 GMT

The queue table needs a date/time column per row.

4. Notification Type belongs on the job, not per-destination
Three types (already in the existing UI as Format):

Full Sheet — attach complete tariff as XLSX
Changes Only — attach only modified rows
Partial Destination — attach only the specific destination
This controls the email attachment and email body template. Should be selected once per job, stored in the push_job record.

5. Pre-validation before job creation
Before writing the job record, run a lightweight check:

For each client:
  - Does the account exist in Sippy? (cached)
  - Does the product-to-tariff mapping resolve? (cached)
  - Does the admin have rate-write access to that tariff?
Return: { valid: 23, invalid: [{accountName, reason}] }

User can review failures before committing the job. Prevents the "997 succeed, 3 fail silently" scenario.

6. Push Job snapshot (audit trail)
On job creation, store a snapshot of:

product_id, product_name, trunk_prefix
client_id, client_name, tariff_id
destination_label, dial_prefix, old_rate, new_rate
effective_at, notification_type

Never query live Sippy data retroactively for the audit log. If the tariff changes later, the snapshot still shows what was sent.

7. Phased status progression
Push History detail should track each stage:

queued → pre_validating → validated → tariff_updating
       → tariff_updated → sbc_updating → sbc_updated
       → email_queued → email_sent → completed
       
       (any stage can → failed, with error + retry count)

SBC stage is a stub for now but the status column should be designed to accommodate it from day one.

8. Recommended phase breakdown
Phase 1 — Job creation (UI only, no background worker yet)

Queue with per-row effective date + price
Notification type selector
Pre-validation summary before submit
Create job record, return job ref
Push History shows job as "queued"
Phase 2 — Background worker

Poll push_jobs WHERE status='queued'
Process via existing action=change path
Update per-client status rows
Retry logic (3 attempts, exponential backoff)
Phase 3 — Email engine

Template per notification type
XLSX attachment generation
Delivery status tracking
Phase 4 — SBC + rollback

SBC update step after tariff confirmation
One-click rollback to snapshot old_rate
The current implementation (direct synchronous push) is correct as a proof-of-concept and can stay as-is for the Rate Manager's "Change Client Rates" modal (small, targeted ops). Send Rate needs the async job architecture described above before it's production-safe.
