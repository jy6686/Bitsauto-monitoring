#337 - Configuration Values Page
What & Why
Reproduce the old BitsAuto Configuration Values page — a central store of operational parameters used across rate management, job scheduling, invoicing, and CDR reconciliation. These values drive business rules throughout the platform (e.g. how many days ahead a rate is effective, alert thresholds, CDR reconciliation delay, invoice settings). Currently these are scattered or hardcoded; this page surfaces them as editable config in a single place.

Done looks like
A Configuration Values page accessible from the platform (under Tools/Settings domain)
5 category tabs: Vendor, A-Z, BSR, Global, Client — matching the old system's tab structure
Each tab shows a table: Configuration name | Unit | Value (inline editable)
Update button saves all changes in the current tab in one request
Filter button allows filtering rows by name within a tab
All rows pre-seeded with sensible defaults matching the old system:
Vendor: Old/Future Effective Date (days), Dial Code Changes Period, Increase Notice Period, Dial Code Length, Rate Increase/Decrease Alert (%), Rate Rounding, Vendor Rate Response Email, Accepted File Size (mb), Acceptable Shortest Duration, Acceptable Pending Increase, Day For Weekly/Fortnightly/Monthly Jobs, Max CDR Reconciliation Delay (hours), Default Domain
Global: System Date/Time formats, File Limit, User Limit, Min Invoice Duration/Amount Difference, Payment Notification Emails
Client: Old/Future Effective Date, Increase Notice Period, Rate Increase/Decrease Alert, Acceptable Pending Increase, Client Rate Notification Email, Standard Rate, Client Rate Rounding, Rate Notifications per Day, Client Invoices Review Required (bool), Auto Send Email Invoices (bool), Send Invoice to KAM (bool), Invoice CC Address, Invoice Difference, Payment Report Duration
A-Z and BSR tabs: empty but present, ready for future population
Admin/management-only write access; support/viewer roles can read but not edit
Out of scope
Wiring these config values into other platform logic (e.g. actually reading Old Effective Date from DB in rate send logic) — this page is the UI and storage layer only; consumption is a separate task
Per-user or per-client overrides — these are global platform settings
Audit log of who changed which value
Steps
DB table — Create configuration_values table via direct SQL (not db:push): columns id, category (vendor/az/bsr/global/client), key, label, unit (nullable), value (text, nullable), value_type (text/number/percent/email/days/bool/domain/mb/hour/float/number). Add to schema.ts.
Seed data — Insert all rows from the three screenshot tabs (Vendor, Global, Client) with their default values via a POST /api/configuration-values/seed endpoint or direct SQL in the migration step.
Backend endpoints — GET /api/configuration-values?category=<tab> returns all rows for a category. PATCH /api/configuration-values accepts an array of { id, value } for bulk update. Admin/management role required for writes.
Frontend page — Build client/src/pages/configuration-values.tsx: 5-tab layout (Vendor / A-Z / BSR / Global / Client), each rendering a table with inline value editing. Update button submits changed rows only. Filter input searches by label client-side.
Navigation — Register the page in app-nav-shell.tsx under an appropriate domain (Tools or Settings). Follow the top-nav rule: all pages go into DOMAINS[], never sidebar-only.
Relevant files
shared/schema.ts
server/routes.ts
client/src/App.tsx
client/src/components/app-nav-shell.tsx
