This is a well-scoped feature set. Here's my architectural recommendation, organized by what to build and in what order — no implementation yet.

Authentication Security Framework — Architecture Recommendation
What BitsAuto Already Has (leverage these)
Capability	Component
Sippy auth rule CRUD	XML-RPC listAuthRules / deleteAuthRule / addAuthRule
WhatsApp alerts	Bhaoo SMS integration (already live)
Automated voice calls	Asterisk AMI + Voice OTP engine
Background scheduler	60s interval job infrastructure
Approval workflow	Dual-approval engine (already built)
Incident tracking	noc_incidents table
No new integrations needed — everything required already exists in the platform.

Recommended Architecture: 3 Layers
Layer 1 — Drift Detection Engine (Background Job)

A new background job runs every 60 seconds per monitored account:

BitsAuto auth_rules DB
        ↕  compare
Sippy listAuthRules (XML-RPC)
        ↓
IF rule exists in Sippy NOT in BitsAuto → DRIFT
IF rule exists in BitsAuto NOT in Sippy → MISSING

Each account has a configurable protection mode:

monitor — log only, no action
alert — log + send alerts
enforce — log + auto-revert + alerts
The key is that monitor is the safe default, with enforce enabled per account explicitly.

Layer 2 — Auto-Revert Engine

Only fires when account is in enforce mode:

Drift Detected
      ↓
deleteAuthRule (Sippy XML-RPC) → remove unauthorized rule
      ↓
Create security incident record (severity: medium/high/critical)
      ↓
Trigger alert pipeline

One important safeguard: a 30-minute emergency override window. If an operator explicitly opens a "change window" in BitsAuto for an account, drift won't trigger auto-revert during that window. This prevents legitimate emergency changes from being auto-reverted while you're actively working.

Layer 3 — Multi-Channel Alert Pipeline

Severity escalation model:

| Severity | Trigger
