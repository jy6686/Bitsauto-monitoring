The document validates that the current BitsEye 2 architecture is semantically correct after the concurrent_snapshots fix:

Suggestions worth acting on
1. Formalize the metric source table as a dev contract
The document provides this table:

Metric	Source	Aggregation
LIVE/DAILY/WEEKLY calls	concurrent_snapshots	MAX(bucket)
ASR	CDR cache	answered/attempted × 100
ACD	CDR cache	total_duration / answered
Minutes	CDR cache	SUM(duration)
Cost	CDR cache	SUM(cost)
Profit	CDR cache	revenue − cost
Suggestion: Add this as a comment block at the top of the BitsEye data-fetch logic (wherever the chart queries live) so any future developer doesn't accidentally reintroduce CDR totals into concurrent graphs. One well-placed comment block prevents the original regression from recurring.

2. NER is not yet implemented — worth adding alongside ASR
The document correctly distinguishes ASR vs NER:

ASR = Answered / Attempted × 100 (strict — only truly answered calls)
NER = (Answered + Busy + Ring-No-Answer) / Attempted × 100 (network-level delivery)
NER is always ≥ ASR and is the metric carriers actually use in interconnect agreements. If the CDR cache includes disposition codes (busy, RONA, etc.), NER is calculable from what's already there.

Suggestion: Add NER as a secondary metric alongside ASR in the quality analytics view. Show both side-by-side. The gap between ASR and NER is itself a diagnostic signal — a large gap means calls are reaching the network but failing at the endpoint, pointing to route quality issues, not connectivity.

3. ACD deserves a fraud/anomaly watch
The document defines ACD correctly. What it doesn't say, but is worth noting: ACD is one of the primary FAS (False Answer Supervision) detection signals. Very short ACD on a specific route or vendor is a red flag.

Suggestion: Where ACD is displayed, add a subtle threshold indicator — if ACD on a specific vendor or route drops below ~8 seconds (configurable), flag it visually. The FAS engine already exists in the platform; ACD-based flagging would complement the existing FAS spike detection.
