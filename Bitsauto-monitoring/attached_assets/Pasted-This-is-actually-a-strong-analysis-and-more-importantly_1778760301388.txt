🧠 1. Core Truth (your system reality check)

You identified the most important point correctly:

Right now BitsAuto is NOT enforcing Sippy auth rules at all.

So the real state is:

⚠️ Current Reality
IP approval exists (or existed), but is now bypassed
Wizard collects auth rules but does NOT enforce or push them
Sippy is created independently
Auth rules are effectively “UI-only data”

👉 Meaning: there is currently no security enforcement layer between BitsAuto and Sippy

This is not dangerous yet, but it is architecturally incomplete.

🔐 2. Correct Target Architecture (final form)

Let’s formalize the system properly:

✔ BitsAuto = Source of Truth (Control Plane)

Responsible for:

IP approval lifecycle
Auth rule creation + validation
Client provisioning logic
Sippy synchronization decisions
✔ Sippy = Stateless Execution Engine

Responsible for:

Accepting validated configuration
Enforcing SIP authentication
Running traffic

Sippy must never contain logic decisions — only applied state.

🚨 3. Critical Correction to Your Current Design

This is the most important refinement:

❌ Wrong assumption in current proposal:

“Wizard creates Sippy account first, then IP approval happens later”

Why this is risky:

Because you now have:

Active SIP account in Sippy
No auth rules yet
Pending IP state in BitsAuto

👉 This creates a partially armed account in telecom terms

Even if no auth rules exist, this is operational confusion and audit risk.

🟢 4. Correct Flow (clean + production-safe)
🔵 PHASE 1 — Wizard (Draft Mode ONLY)

Wizard should ONLY:

Create Company
Create Client (BitsAuto DB only)
Store configuration drafts
Store IPs as PENDING

❌ NO Sippy call here

🔵 PHASE 2 — Activation (Explicit Step)

Add a single action:

“Provision Client to Sippy”

This action:

Creates Sippy account
Sets NO AUTH RULES
Marks account = PROVISIONED
🔵 PHASE 3 — IP Approval Gate
IPs submitted → PENDING
Admin approves → APPROVED
Only approved IPs become eligible for auth rules
🔵 PHASE 4 — Auth Rule Creation (STRICT CONTROL)
Hard rule:
IF IP.status != APPROVED → BLOCK

When allowed:

BitsAuto creates auth rule
Immediately pushes to Sippy via XML-RPC
Stores mapping locally
🔵 PHASE 5 — Sync Enforcement

Any change in BitsAuto triggers Sippy sync:

IP revoked → disable auth rule in Sippy
IP updated → update Sippy rule
Client disabled → remove/disable all auth rules
🔐 5. The Real Security Model (important upgrade)

What you're building is actually:

✔ “Policy-driven SIP provisioning system”

That means:

Layer	Responsibility
BitsAuto DB	Truth
Approval Engine	Gatekeeper
Sync Engine	Sippy updater
Sippy	Execution only

❌ Never allow this state:

Sippy account exists + no policy enforcement in BitsAuto

Because then:

someone can manually modify Sippy later
or legacy scripts can inject auth rules
or credentials leak becomes catastrophic
