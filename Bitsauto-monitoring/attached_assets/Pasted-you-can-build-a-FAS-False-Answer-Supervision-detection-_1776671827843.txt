Let me break this down practically for your environment 👇

✅ What You Want to Detect (FAS Scenarios)

You already defined it correctly:

🔍 Detection Cases
Call answered (SIP 200 OK) but no RTP audio
Early answer (200 OK before normal ring duration)
Dead air / silence after answer

These are classic revenue leakage / fraud cases in VoIP.

🧠 How It Works (Core Logic)
1. Capture SIP Signaling (From Sippy Soft)

From Sippy you can get:

Call-ID
INVITE timestamp
180 Ringing
200 OK (Answer time)

👉 This tells you when call was “answered”

2. Capture RTP Media (Very Important)

This is the real challenge.

You need:

RTP packet flow (start time, packet count)
Audio presence (not just packets—actual voice)

✔ Options:

Use RTP proxy / media proxy logs
Use pcap / packet capture
Use RTCP stats (if enabled)
3. Detection Logic (Core Engine)
📊 Case 1: No RTP after Answer
IF (200 OK received)
AND (No RTP packets within 2–3 seconds)
→ Flag as FAS
📊 Case 2: Early Answer
IF (200 OK < 1–2 sec after INVITE)
→ Suspicious
📊 Case 3: Silence Detection
IF (RTP present BUT audio level ~ zero for X seconds)
→ FAS / Dead Air
