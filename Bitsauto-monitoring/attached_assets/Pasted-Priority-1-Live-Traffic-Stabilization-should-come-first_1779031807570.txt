Priority 1 — Live Traffic Stabilization should come first.

Not because it is the most exciting feature, but because it is now the foundation underneath:

Live Traffic
Entity Verdict
Routing projections
future SLA reporting
executive analytics

If the live telemetry layer drifts or behaves inconsistently under load, every higher-level decision layer becomes untrustworthy.

Recommended Immediate Plan
Phase A — Telemetry Validation Harness

Before changing architecture, build a controlled validation layer around the existing engine.

Validate:

Area	What to test
Rolling ASR accuracy	1m/5m/15m windows match raw CDR truth
Snapshot drift	repeated recalculation consistency
Sparse traffic handling	1–3 call windows don't spike falsely
Delta correctness	↑↓ indicators only move on meaningful changes
Memory growth	long-running process stability
WS reconnect	browser reconnect after disconnect/network flap
Multi-client fanout	multiple dashboards subscribed simultaneously
Recompute latency	snapshot generation time under large cache

This is operational hardening, not feature work.

Simulator

Priority 3 — Executive Layer (after stabilization)

The advisor is right:
you now finally have enough trustworthy telemetry for executive reporting.

But executive dashboards before stabilization are dangerous because:

executives trust visuals immediately
they will not understand telemetry caveats
inconsistent live metrics destroy confidence fast

So:

stabilize
unify workflows
then expose executive reporting

That order is correct.

One additional recommendation

Before building the Investigate workflow, establish a canonical entity identity standard.

Right now you have:

accountName
clientName
vendor
connection
routing group
iConnection
accountId

The drawer/workflow layer will become messy unless every entity resolves through a single identity normalization layer.

Example:

entityType: "connection"
entityId: "123"
canonicalName: "XYZ Pakistan"
vendorName: "XYZ Telecom"

This will save enormous complexity later.
