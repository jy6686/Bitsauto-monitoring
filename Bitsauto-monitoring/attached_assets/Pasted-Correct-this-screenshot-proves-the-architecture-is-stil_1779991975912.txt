The key architectural thread running through all three documents is the same: client identity normalization is the missing foundation — without it, invoices, DMR, reconciliation, and disputes each resolve identity differently, causing the fragmentation you're describing.

Based on the priority ordering laid out (T001 → T002 → T003 → T004), the logical next step would be:

T001 — Client Identity Layer
A client_identity_map table binding i_account → sippy_username → billing_name → display_name → account_manager → external_ref. This becomes the canonical source for everything: invoices, DMR, reconciliation, disputes, AI assurance alerts.

T002 — Finance Cockpit
Now possible because the data layer is stable: DMR fallback works, reconciliation is account-bound, AI assurance exists. This is the unified workspace (KPI strip + collections queue + revenue assurance grid + AI queue) that turns the finance modules from "many pages" into one operational workspace.
