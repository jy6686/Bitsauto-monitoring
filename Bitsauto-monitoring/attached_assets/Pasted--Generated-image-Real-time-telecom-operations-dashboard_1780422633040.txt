The analysis is accurate. Let me validate each observation against the code to give you a precise picture.

On NER = 0.0% per row but 23.4% at the total:

This is a specific mapping inconsistency, not a general calculation failure. The portal path (toReportRow in routes.ts, line ~4033) explicitly sets nerPct: r.asr for each row — so individual rows should show NER equal to ASR. The fact that rows show 0.0% while the total shows 23.4% means one of two things: either the frontend is reading a different field for per-row NER display than for the total row, or calcTotal aggregates NER from a different source than toReportRow populates. The fix is a one-line mapping correction, not a data architecture change.

On FAS = 0.0% everywhere:

This is by design in the current portal path — the code explicitly sets fasRate: 0 with the comment "portal has no per-call duration distribution." The document's assessment is correct: FAS requires per-call SIP response codes or short-duration pattern detection, neither of which is available from /asr_acd.php's aggregated output. Showing 0.0% is misleading — "—" or "N/A" is the right representation.

On the recommendation to not display 0% NER:

Strongly agree. An operator reading 0% NER could interpret it as "0% of calls reached the network destination" (catastrophic) or dismiss it as a rendering glitch. Either way it erodes trust in the platform. Showing "—" with a tooltip ("NER requires per-call data — not available from aggregated report") is far safer than a false zero.

Priority assessment matches the code reality:

Priority	Issue	Complexity
P2-A	NER row/total mapping inconsistency	Small — likely one field reference fix
P2-B	Hide FAS and NER when unavailable from portal source	Small — conditional render
P2-C	Real FAS from CDR-cache path (has per-call durations)	Medium — already partially exists
P2-D	Metric provenance tooltip (Derived vs Native)	Small — UI label only
The current platform state is healthy at the telemetry layer. P2-A and P2-B are low-risk corrections that just prevent misleading zeros from appearing in an otherwise accurate report.
